# Sira 贡献指南

## 🌟 欢迎参与贡献！

**Sira** 是一个学生学习项目，我们欢迎所有热爱学习和开源技术的同学参与贡献！

这个指南的目标是：

- 🆕 **鼓励新贡献**：让初学者也能轻松参与
- 🔄 **保持参与度**：让贡献者持续投入项目
- 🚀 **简化流程**：避免不必要的繁琐步骤
- 📖 **透明决策**：让贡献者清楚如何参与决策

## 👥 角色定义

- **贡献者 (Contributor)**：任何创建或评论Issue、PR的人
- **维护者 (Maintainer)**：负责项目日常维护的同学
- **核心贡献者**：对项目有重要贡献的同学

## 📝 提交问题 (Issues)

遇到任何问题或有好的想法？欢迎提交Issue！

### 🐛 问题报告

- **技术问题**: 遇到bug、错误或异常行为
- **功能建议**: 有新的功能想法或改进建议
- **学习疑问**: 对代码或架构有不明白的地方
- **文档问题**: 发现文档错误或需要补充

### 📋 Issue 内容建议

```
标题: [简明描述问题]

描述:
- 问题现象
- 复现步骤
- 期望结果
- 环境信息 (OS, Node.js版本等)

标签: bug/feature/question/docs
```

### 💬 沟通礼仪

- 使用友好的语言，尊重每位贡献者
- 遇到分歧时，保持理性讨论
- 所有参与者必须遵守[行为准则](CODE_OF_CONDUCT.md)

## 🗺️ 项目规划

### 学习路线图

我们根据学习进度规划项目发展：

1. **基础阶段**: AI网关核心功能实现
2. **进阶阶段**: 性能优化和监控完善
3. **扩展阶段**: 新功能开发和生态建设

### 功能优先级

- 🔥 **高优先级**: 影响核心功能的问题
- 🟡 **中优先级**: 性能优化和用户体验改进
- 🔵 **低优先级**: 新功能和扩展特性

## 🤝 如何贡献代码

### 🔄 Pull Request 流程

所有代码变更都通过Pull Request (PR) 提交，包括：

- 代码修改、bug修复
- 新功能开发
- 文档更新
- 配置调整

### 📋 PR 要求

一个好的PR应该包含：

- ✅ **清晰的标题**: 简要描述变更内容
- ✅ **详细描述**: 解释为什么要改，改了什么
- ✅ **测试验证**: 相关的测试用例
- ✅ **文档同步**: 如涉及API变更，要更新文档

### 🧪 测试要求

- 在 `test/` 目录下添加相应的测试
- 确保所有现有测试通过
- 运行 `npm test` 验证

### 📖 文档要求

- 更新相关文档
- 如有API变更，更新 `API-DOCS.md`
- 添加必要的注释

### 🔍 代码审查

- PR提交后会自动触发CI检查
- 维护者会在合理时间内进行审查
- 审查通过后才能合并

### ⏱️ 审查时间

- 简单PR: 24小时内审查
- 复杂PR: 2-3天内完成审查
- 节假日: 会适当延长审查时间

## 🎓 贡献者成长

### 🌱 新手贡献者

- **从简单开始**: 先修复小bug、改进文档
- **学习机会**: 通过贡献学习最佳实践
- **导师制度**: 有经验的贡献者会指导新人

### 🏆 核心贡献者

积极贡献的同学会被邀请成为核心贡献者，获得：

- 直接推送权限
- 项目决策参与权
- 维护者职责

### 🤝 合作准则

- **共识优先**: 通过讨论达成共识
- **妥协精神**: 为项目整体利益做出让步
- **开放沟通**: 保持透明的决策过程

## 🚀 快速开始贡献

1. **Fork项目** 到你的GitHub账户
2. **克隆代码** 到本地: `git clone https://github.com/YOUR_USERNAME/ai-gateway.git`
3. **创建分支** : `git checkout -b feature/amazing-feature`
4. **提交更改** : `git commit -m 'Add amazing feature'`
5. **推送分支** : `git push origin feature/amazing-feature`
6. **创建PR** : 在GitHub上提交Pull Request
